# Weathering 跨语言迁移评估手册（Python / Java）

> 问题边界：将“影响游戏流程的核心逻辑”作为迁移重点；  
> Shader、渲染后处理、纯 QoL 视觉特效等可视为次要/可降级项。

## 1. 结论先行

- **可迁移**：是的，核心玩法逻辑可迁移到 Python/Java。  
- **推荐路径**：优先迁移为“逻辑内核（Headless Simulation Core）”，而非一次性迁移完整 Unity 客户端。  
- **语言选择建议**：
  - 若目标是工具化验证、原型迭代、脚本化分析：优先 Python。
  - 若目标是长期运行时、性能与工程化发布：优先 Java（或 Kotlin/JVM）。

---

## 2. 可迁移性分层评估

## 2.1 高可迁移层（建议优先）

1. **世界/地图生成规则**  
   包括 hash 驱动生成、地图层级 key 规则、默认地块逻辑。
2. **经济与生产链逻辑**  
   输入/输出、库存容量、工厂运行条件、连线关系。
3. **状态系统**  
   `Value/Ref/Inventory` 的状态推进与结算。
4. **存档结构与回放**  
   JSON 序列化协议、版本迁移逻辑。

这些部分本质是“确定性状态机 + 数据转换”，与 Unity 引擎耦合度低。

## 2.2 中可迁移层（需抽象）

1. **地图交互行为**（Tap/构建/拆除）
2. **时间推进与调度**
3. **事件系统与提示系统**

可迁移，但需要先定义平台无关接口。

## 2.3 低优先迁移层（可延后）

1. Shader/材质/渲染特效
2. Unity Scene 与 Tilemap 表现层
3. 现有 UI 预制件结构

题设已说明 QoL 视觉项可忽略，这使迁移风险显著下降。

---

## 3. Python 与 Java 对比

## 3.1 Python 路线

### 优势

- 开发快，便于“规则复刻 + 校验脚本 + 设计回归测试”。
- 数据处理友好，适合生成平衡性报告。
- 与现有 `PlanetInfo` 工具生态契合。

### 风险

- 大规模实时模拟性能一般。
- 长期多人协作的大型工程治理成本较高（类型约束弱）。

### 适合场景

- 规则验证器、离线模拟器、策划工具链、平衡性自动评估。

## 3.2 Java 路线

### 优势

- 性能稳定、类型系统强、工程化与模块化成熟。
- 更适合作为长期“服务端模拟核心”或跨平台桌面 runtime。

### 风险

- 初期开发速度通常慢于 Python。
- 与现有 Unity C# 代码语义映射要做较多样板层。

### 适合场景

- 长期维护的逻辑核心、服务端权威模拟、多人联机后端。

## 4. 关键技术可行性点

## 4.1 随机与确定性

项目大量依赖哈希与确定性生成，迁移必须保证：

- 相同输入在目标语言输出一致；
- 溢出、整数位宽、取模负值行为统一；
- 随机种子算法与顺序固定。

建议先建立“跨语言黄金样例集（Golden Cases）”。

## 4.2 类型系统映射

C# 的 `Type` 反射机制需要映射为跨语言“注册表 + 稳定 ID”机制：

- 不以运行时类名作为持久化主键；
- 用 `content_id` 映射到具体实现。

这一步也是模组化与跨语言复用的共同前提。

## 4.3 存档兼容

旧存档如果存的是 CLR 类型名，迁移时要提供：

1. 别名映射表（旧类型名 -> 新 content_id）
2. 版本迁移器
3. 回退策略（缺失模组/缺失类型时的降级行为）

## 4.4 运行时接口隔离

抽离 3 层：

- `Simulation Core`（纯逻辑）
- `Platform Adapter`（输入、文件、时钟）
- `Presentation`（Unity/其他前端）

这样可以做到“同一逻辑核心，多前端复用”。

---

## 5. 推荐迁移策略（务实版）

## 阶段 A：先做逻辑镜像而非替换

- 在 Python 或 Java 实现最小闭环：`Value/Ref/Inventory + 地图 key 解析 + 基础生产链`。
- 与现有 C# 行为对拍（黄金样例自动测试）。

## 阶段 B：建立协议与注册规范

- 定义统一 `content_id`、`save_schema`、`manifest`。
- C# 与新语言同时遵守同一协议。

## 阶段 C：逐步扩大覆盖率

- 从“宇宙/地图生成 + 经济系统”扩到“建筑行为 + 事件系统”。
- 视觉层仍由 Unity 承担或最小实现。

## 阶段 D：决定最终架构

两种可落地方向：

1. **Unity 前端 + 外部逻辑核心（Python/Java）**
2. **完全新客户端（不依赖 Unity）**

建议先走方向 1，成本更低、回滚更容易。

---

## 6. 风险清单与缓解

1. **行为不一致风险**：通过黄金样例 + 回放测试缓解。  
2. **性能风险**：先做基准测试（地图规模、tick 频率、连线数量）。  
3. **模组生态割裂**：统一 manifest 与 content_id 协议。  
4. **迁移范围失控**：严格“逻辑优先、表现后置”。

---

## 7. 最终建议

- 这项目**非常适合**做“跨语言逻辑核心迁移”，因为其玩法核心是数据驱动、规则驱动、确定性驱动。  
- 在你给出的前提（Shader/QoL 可忽略）下，迁移复杂度从“整机迁移”降低为“规则引擎迁移”，可行性显著提高。  
- 若你要快速产出：先 Python；若你要长期产品化：优先 Java。  
- 无论选哪种语言，第一优先事项都应是：**先标准化内容 ID、注册中心与存档协议**。
